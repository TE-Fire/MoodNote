# 跨域配置详解

## 概述

跨域（Cross-Origin Resource Sharing，CORS）是浏览器的一种安全机制，用于限制从一个源加载的文档或脚本如何与来自另一个源的资源进行交互。当前端应用和后端 API 部署在不同域名时，必须配置跨域支持。

## 一、WebMvcConfigurationSupport 与 WebMvcConfigurer 的区别

### 1.1 继承 vs 实现

| 特性 | WebMvcConfigurer | WebMvcConfigurationSupport |
|------|------------------|---------------------------|
| 使用方式 | 实现接口 | 继承类 |
| 配置方式 | 选择性覆盖方法 | 重写模板方法 |
| 默认配置 | 保留 Spring Boot 默认配置 | 完全自定义配置 |
| 适用场景 | 增量配置 | 完全控制配置 |

### 1.2 WebMvcConfigurer（推荐使用）

```java
@Configuration
public class MyConfig implements WebMvcConfigurer {
    
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        // 仅配置跨域，其他配置保持默认
        registry.addMapping("/api/**").allowedOrigins("*");
    }
}
```

**特点**：
- 基于 Java 8 的接口默认方法
- 可以选择性地覆盖需要的配置
- **保留 Spring Boot 的自动配置**
- 适合大多数场景

### 1.3 WebMvcConfigurationSupport（谨慎使用）

```java
@Configuration
public class MyConfig extends WebMvcConfigurationSupport {
    
    @Override
    protected void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**").allowedOrigins("*");
    }
}
```

**特点**：
- 需要继承类
- **会完全替换 Spring Boot 的自动配置**
- 需要手动配置视图解析器、静态资源等
- 适合需要完全控制 MVC 配置的场景

### 1.4 关键区别总结

```
使用 WebMvcConfigurer
    │
    ├─ 保留 Spring Boot 默认配置
    ├─ 仅覆盖需要的配置
    └─ 推荐用于大多数场景

使用 WebMvcConfigurationSupport
    │
    ├─ 完全替换默认配置
    ├─ 需要手动配置所有内容
    └─ 仅用于特殊需求场景
```

> **注意**：如果使用 `@EnableWebMvc` 注解，也会完全替换 Spring Boot 的自动配置，效果类似继承 `WebMvcConfigurationSupport`。

## 二、跨域配置类的作用

### 2.1 什么是跨域问题

当浏览器从 `http://localhost:3000`（前端）请求 `http://localhost:8080`（后端）时，浏览器会认为这是跨域请求：

```
前端域名: http://localhost:3000
后端域名: http://localhost:8080

协议相同: http = http ✓
域名相同: localhost = localhost ✓
端口不同: 3000 ≠ 8080 ✗ → 跨域！
```

### 2.2 配置类的核心作用

```java
@Configuration
public class CorsConfig implements WebMvcConfigurer {
    
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
                .allowedOrigins("*")
                .allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS")
                .allowedHeaders("*")
                .exposedHeaders("Content-Type", "Authorization")
                .allowCredentials(true)
                .maxAge(3600);
    }
}
```

### 2.3 配置项详解

| 方法 | 作用 | 示例 |
|------|------|------|
| `addMapping()` | 指定允许跨域的路径模式 | `/api/**` |
| `allowedOrigins()` | 允许的来源域名 | `*` 或具体域名 |
| `allowedMethods()` | 允许的 HTTP 方法 | `GET, POST, PUT, DELETE` |
| `allowedHeaders()` | 允许的请求头 | `*` |
| `exposedHeaders()` | 暴露给前端的响应头 | `Authorization` |
| `allowCredentials()` | 是否允许携带凭证 | `true` |
| `maxAge()` | 预检请求缓存时间（秒） | `3600` |

## 三、实现验收标准的原理

### 3.1 验收标准回顾

1. 跨域配置类创建成功
2. **配置了允许的源、方法、头部等**
3. **预检请求缓存时间设置合理**
4. 前端应用能够正常访问后端 API

### 3.2 如何实现"允许的源、方法、头部"

#### 1. 允许的源（allowedOrigins）

```java
.allowedOrigins("*")
```

**作用**：允许所有域名访问（开发环境）

**生产环境建议**：
```java
.allowedOrigins("https://www.moodnote.com", "https://moodnote.com")
```

**原理**：在响应头中添加 `Access-Control-Allow-Origin`

#### 2. 允许的方法（allowedMethods）

```java
.allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS")
```

**作用**：允许指定的 HTTP 方法

**原理**：在响应头中添加 `Access-Control-Allow-Methods`

#### 3. 允许的头部（allowedHeaders）

```java
.allowedHeaders("*")
```

**作用**：允许前端发送任意请求头

**原理**：在响应头中添加 `Access-Control-Allow-Headers`

### 3.3 如何实现"预检请求缓存时间"

#### 什么是预检请求（Preflight Request）

当发起复杂跨域请求时，浏览器会先发送一个 `OPTIONS` 请求进行预检：

```
┌─────────────────────────────────────────────────────────────┐
│  浏览器：先发送 OPTIONS 请求检查服务器是否允许跨域          │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  服务器：返回允许的源、方法、头部等信息                      │
│  Access-Control-Allow-Origin: *                            │
│  Access-Control-Allow-Methods: GET, POST, PUT, DELETE      │
│  Access-Control-Max-Age: 3600                             │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  浏览器：缓存预检结果 1 小时，期间不再发送 OPTIONS 请求       │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  浏览器：发送实际的 GET/POST 请求                           │
└─────────────────────────────────────────────────────────────┘
```

#### 设置缓存时间

```java
.maxAge(3600) // 单位：秒
```

**作用**：设置预检请求结果的缓存时间为 1 小时

**原理**：在响应头中添加 `Access-Control-Max-Age: 3600`

**为什么需要缓存**：
- 减少 OPTIONS 请求次数
- 提高性能
- 避免频繁的预检请求影响用户体验

### 3.4 完整流程图解

```
前端请求 (POST /api/login)
        │
        ▼
┌─────────────────────────────────────────────────────────────┐
│                    Spring MVC 拦截器                        │
│  CorsInterceptor: 检查 Origin 头                          │
│  判断是否为跨域请求                                        │
└─────────────────────────────────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────────────────────────────────┐
│              是否为预检请求 (OPTIONS)?                       │
│                      │                                     │
│          ┌───────────┴───────────┐                        │
│          ▼                       ▼                        │
│      是 (OPTIONS)            否 (GET/POST)                │
│          │                       │                        │
│          ▼                       ▼                        │
│  返回 CORS 响应头          继续处理业务逻辑                 │
│  Access-Control-*          并在响应中添加 CORS 头          │
└─────────────────────────────────────────────────────────────┘
```

## 四、跨域请求的类型

### 4.1 简单请求 vs 预检请求

| 类型 | 条件 | 示例 |
|------|------|------|
| **简单请求** | GET/POST/HEAD + 简单头 | `GET /api/users` |
| **预检请求** | 其他方法或复杂头 | `PUT /api/users/1` |

**简单请求条件**：
1. 方法是 GET、POST、HEAD 之一
2. 请求头只包含：Accept、Accept-Language、Content-Language、Content-Type（仅限 application/x-www-form-urlencoded、multipart/form-data、text/plain）

### 4.2 凭证请求

当需要携带 Cookie 或 Authorization 头时：

```java
.allowCredentials(true)
```

**注意**：当设置 `allowCredentials(true)` 时，`allowedOrigins` 不能设置为 `"*"`，必须指定具体域名。

## 五、生产环境最佳实践

### 5.1 配置具体域名

```java
@Override
public void addCorsMappings(CorsRegistry registry) {
    registry.addMapping("/api/**")
            .allowedOrigins(
                "https://www.moodnote.com",
                "https://moodnote.com",
                "https://app.moodnote.com"
            )
            .allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS")
            .allowedHeaders("*")
            .exposedHeaders("Authorization")
            .allowCredentials(true)
            .maxAge(3600);
}
```

### 5.2 使用环境变量配置

```java
@Value("${cors.allowed-origins}")
private String[] allowedOrigins;

@Override
public void addCorsMappings(CorsRegistry registry) {
    registry.addMapping("/api/**")
            .allowedOrigins(allowedOrigins)
            // ...
}
```

application.yml：
```yaml
cors:
  allowed-origins:
    - https://www.moodnote.com
    - https://app.moodnote.com
```

### 5.3 使用 Spring Security 的 CORS 配置

如果项目使用 Spring Security，需要在 SecurityConfig 中配置：

```java
@Configuration
public class SecurityConfig {
    
    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            .cors(cors -> cors.configurationSource(corsConfigurationSource()))
            // ...
        return http.build();
    }
    
    @Bean
    public CorsConfigurationSource corsConfigurationSource() {
        CorsConfiguration config = new CorsConfiguration();
        config.setAllowedOrigins(Arrays.asList("https://www.moodnote.com"));
        config.setAllowedMethods(Arrays.asList("GET", "POST", "PUT", "DELETE"));
        config.setAllowedHeaders(Arrays.asList("*"));
        config.setAllowCredentials(true);
        config.setMaxAge(3600L);
        
        UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
        source.registerCorsConfiguration("/api/**", config);
        return source;
    }
}
```

## 六、常见问题与解决方案

### 6.1 跨域配置不生效

**问题**：配置了 CorsConfig 但仍然报跨域错误

**排查步骤**：

1. **检查配置类是否被 Spring 扫描**：确保有 `@Configuration` 注解且包路径正确
2. **检查请求路径是否匹配**：确认请求路径符合 `addMapping` 的模式
3. **检查是否有其他过滤器拦截**：Spring Security 等可能会先拦截请求
4. **检查响应头**：使用浏览器开发者工具查看响应头是否包含 `Access-Control-*`

### 6.2 携带凭证时报错

**错误信息**：
```
The value of the 'Access-Control-Allow-Origin' header in the response must not be the wildcard '*' when the request's credentials mode is 'include'.
```

**解决方案**：
```java
// 错误：不能同时使用 * 和 allowCredentials(true)
.allowedOrigins("*")
.allowCredentials(true)

// 正确：指定具体域名
.allowedOrigins("https://www.moodnote.com")
.allowCredentials(true)
```

### 6.3 自定义响应头无法获取

**问题**：后端返回了自定义响应头，但前端无法获取

**解决方案**：
```java
.exposedHeaders("X-Custom-Header", "Authorization")
```

## 七、总结

### 核心要点

| 要点 | 说明 |
|------|------|
| **WebMvcConfigurer** | 推荐使用，保留默认配置 |
| **WebMvcConfigurationSupport** | 完全自定义，谨慎使用 |
| **allowedOrigins** | 允许的来源域名 |
| **allowedMethods** | 允许的 HTTP 方法 |
| **allowedHeaders** | 允许的请求头 |
| **maxAge** | 预检请求缓存时间 |
| **allowCredentials** | 是否允许携带凭证 |

### 学习建议

1. **理解同源策略**：浏览器的安全机制
2. **区分简单请求和预检请求**：不同的处理方式
3. **生产环境配置**：不要使用 `*`，指定具体域名
4. **结合 Spring Security**：需要额外配置
